# 文档写作规范

适用于本仓库 `docs/` 下由团队维护的 Markdown 与配套附件。**代码即真相**：文档只描述「范围、约束、操作步骤、与代码的对应关系」，不重复粘贴大段可过时实现。

---

## 1. 文档类型与必选结构

### 1.1 设计 / 方案类（如知识库、子系统设计）

| 章节顺序 | 内容要求 |
|----------|----------|
| 1. 目标与非目标 | 写清做 / 不做，避免范围 creep |
| 2. 背景与术语 | 可选；新名词在此定义一次 |
| 3. 方案说明 | 架构、数据流、与现有模块关系；可用表格、列表，少用长散文 |
| 4. 接口或数据契约 | 路径、字段、枚举与代码一致；可写「见 Prisma 模型 xxx」代替重复 schema |
| 5. 风险、约束、合规 | 微信、版权、医疗表述等 |
| 6. 迭代与状态 | 已实现 / 待办用表格标注，并写日期或版本一句即可 |

**避免**：无标题长文、无锚点的「一段到底」、与 `superpowers/specs` 重复的全量需求复述（应链接 spec）。

### 1.2 运维 / 操作类（如部署）

| 章节顺序 | 内容要求 |
|----------|----------|
| 1. 前置条件 | 环境版本、账号、权限 |
| 2. 配置说明 | 环境变量表：名称、是否必填、含义、示例值（无秘密） |
| 3. 操作步骤 | 编号步骤；每条命令注明**工作目录**（仓库根 / `backend/`） |
| 4. 验证与排错 | 如何确认成功；常见错误与处理 |
| 5. 相关路径 | 指向代码或 Compose 文件路径一行即可 |

**避免**：复制粘贴未验证的命令、假定的域名/IP 当唯一真理。

### 1.3 产品 / 商业备忘（`product/`）

| 章节顺序 | 内容要求 |
|----------|----------|
| 1. 文档目的与读者 | 一句 |
| 2. 正文 | 预算、竞品、叙事等按材料自身逻辑 |
| 3. 与实现的边界 | 明确「非接口契约」；行为以 spec + 代码为准 |

**避免**：把备忘当需求基线；与 `superpowers` 冲突时不写死结论，应注明「以 spec 为准」。

### 1.4 仓库文档首页（`docs/README.md`）

- **唯一索引页**：目录约定 + **完整文件索引表** + 自动化路径说明 + 指向本规范的链接。
- **不在**各子目录再维护一份「迷你首页」；子目录仅放主题文档与素材。

---

## 2. 格式与排版

- **标题**：从 `##` 起用（`#` 仅文档总标题）；同级标题语义并列，**不要跳级**（如 `##` 后直接 `####`）。
- **列表**：步骤用有序列表；并列项用无序列表；避免嵌套过深（建议 ≤2 层）。
- **表格**：环境变量、对照、状态用表；列不宜过多（≤6 列易读）。
- **代码与命令**：使用带语言的 fenced code block；Shell 写明 `cd` 上下文。
- **链接**：优先相对路径（如 `engineering/deployment.md`）；链到仓库外须完整 URL。
- **语言**：说明性文字以**中文**为主；**标识符**（API 路径、env、类型名、文件名）与代码一致，用英文反引号包裹。

---

## 3. 内容质量

- **可验证**：涉及「已支持 / 未支持」的陈述，应对照路由、Prisma 或测试；不确定写「待确认」。
- **可维护**：大改实现时同步改文档；删除功能时删对应段落而非只标废弃。
- **合规**：涉及健康、用药、诊断的表述须与产品免责声明一致，避免包治、确诊等用语。
- **秘密**：不在文档中写真实密钥、token；示例用占位符。

---

## 4. 禁止事项

- 为每个子目录单独再写一份冗长 `README.md` 重复索引（**索引只维护 `docs/README.md`**）。
- 在正文里堆叠大量 `@路径` 或重复引用同一附件；一处入口表即可。
- 把未纳入版本控制的个人笔记当正式规范路径。
